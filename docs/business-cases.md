## 2. Product Description (Описание продукта)

Разрабатываемый продукт - решение для оптимального управления городской полицией.

Продукт решает cледующие основные задачи: 
- Управление полицией. 
- Этичная, непредвзятая оценка граждан с точки зрения способности к совершению преступлений.

Вследствие этого:
- Безопасность на улицах городов повышается.
- Повышается инвестиционная привлекательность городов.
- Повышается удовлетворённость граждан работой полиции.

## 3. Business Context (Бизнес-контекст)

- Продукт предназничен для управления городской полицией. То есть он применяется в сфере государственного управления. 

- Продукт будет закупаться городскими властями с целью экономии средств на содержании полиции, установления порядка в городах и, как следствие, повышения инвестиционной привлекательности городов.

- Используется городской полицией для оптимизации её работы.

- Продукт является коммерческим решением и поставляется по подписке.

## 4. Product Objectives (Цели продукта)

### 4.1 Цели разработки продукта.

- Основная цель разработки продукта - поставка его на условиях подписки заказчику и извелечение прибыли.

Для этого выделим фазы, которые будут вести к готовности продукта.

#### Фаза 1. Начальная стадия (Inception)

Включает определение следующих вещей:

- Видение и границы проекта.
- Экономическое обоснование (business case)
- Основные требования, ограничения и ключевую функциональность продукта.
- Базовую версию модели прецедентов.
- Оценку рисков.

В этой фазе создаются документы:

- Vision Document
- Use Cases Document
- Software Requirements Specification Document
- Risks List Document
- Software Development Plan Document
- Glossary
- Business Case Document

#### Фаза 2. Уточнение (Elaboration)

Цель - Детализация архитектуры системы, описание стратегии тестирования.

В этой фазе создаются следующие документы:

- Software Architecture Document
- Test Plan Document

#### Фаза 3. Построение (Construction)

##### Итерация 1: 

- Разработка основной возможности продукта - непредвзятой оценки способности граждан совершать предступления.
- Демонстрация потенциальным заказчикам раннего прототипа.
- После одобрения потенциальными заказчиками, предоставление скидок при раннем заказе продукта.

В этой итерации создается демо-версия продукта.

##### Итерация 2:

- Разработка MVP продукта, поставка MVP первым заказчикам.

В этой итерации создается версия продукта 1.0.0

##### Итерация 2+:

- Работа отдела продаж, выявление нового требуемого функционала.
- Разработка нового функционала системы.
- Поставка функционала по мере его готовности.
- Продажа продукта с новым функционалом новым заказчикам.

В последующих итерациях создаются версии продука > 1.0.0. Подробно значение разных версий описано в SDP, п.4.2.3

#### Фаза 4. Внедрение (Transition)

##### Итерация 1
Цель - создание финальной версии продукта и поставка его заказчику.
В этой фазе продукт готовится к развертыванию в production среде и развёртывается.

##### Итерация 2+
В последующих итерациях в production среде развертываются новые версии продукта.

### 4.3 Предварительная оценка рисков.

Подробно риски оценены в документе Risk List. 

Кратко приведем главные риски с точки зрения бизнеса:
1.  Средства будут вложены, а продукт никто не купит.

    - Этот недостаток мы пытаемся нивелировать ранней поставкой MVP продукта и тесным взамиодействием с потенциальным заказчиком, направленным решением его проблем.

2. Продукт будет пользоваться спросом только у первых заказчиков.

    - Продукт поставляется по подписке, поэтому даже небольшое число постоянных клиентов позволяет проекту вернуть средства, затраченные на разработку и получить прибыть.
    
    - Продукт постоянно развивается и получает новые возможности, чтобы соответствовать требованиям новых заказчиков.

## 5. Financial Forecast (Финансовый прогноз)

Подробно стоимость разработки расчитана в Software Development Plan, п. 4.2.4. Будем пользоваться полученными в этом документе числами.

### Затраты до первой поставки продукта

#### Фаза 1. Начальная стадия.

**Длительность фазы 1: 18 недель**

На начальной стадии разрабатывается документация проекта. Для разработки документации требуются два специалиста: Software developer-manager и Software architect-manager. Исходя из оклада 3 бутылки пива в неделю, на эту фазу уйдёт 102 бутылки пива.

#### Фаза 2. Уточнение. 

**Длительность фазы 2: 4 недели**

В фазе уточнения разрабатывается документация, в основном техническая. Для этого, как и в начальной стадии, требуется 2 человека. На эту фазу уйдёт 24 бутылки пива.

#### Фаза 3. Построение. 

**Длительность итерации 1: 2 недели**
Следуя плану, сформированному в п. 4, в итерации 1 фазы 3 прототип продукта разрабатывается силами небольшого отдела и финансируется Equilibrium inc. При успешном завершении фазы 3 и получении предварительных заказов, с этого момента проект выходит на самоокупаемость и финансируется за счет предварительных продаж. В этой итерации к команде добавляется ещё один сотрудник - Software developer. Стоимость этой итерации - 18 бутылок пива

**Длительность итерации 2: 6 недель**
Разрабатывается MVP продукта, отделы продаж используют прототип для демонстрации и получают больше предзаказов.
Стоимость итерации - 54 бутылки пива. Финансирование ведется из средств, полученных через предзаказы.

Итоговая стоимость фазы - 72 бутылки пива.

#### Фаза 4. Внедрение. 

**Длительность фазы 4: 1 неделя**

После завершения работы надо продуктом, он тестируется в production среде и поставляется заказчику.
Стоимость этой фазы - 9 бутылок пива.

--------------------------------------------------------
Итог: Стоимость разработки продукта - ~200 бутылок пива.

### Затраты после первой поставки продукта

После выпуска продукта затраты на его поддержку и дальнейшую разработку составляют 36 бутылок пива в месяц.

### Окупаемость проекта

Среднему потребителю - небольшму европейскому государству - продукт постваляется по подписке. Стоимость подписки 200 бутылок пива в год. Компания планирует обслуживать по меньшей мере 5 заказчиков. Таким образом, чистая прибыль после выхода на окупаемость составит 200 * 5 - 36 * 12 = 568 бутылок пива в год. Окупаться проект будет в течение 200/(568/12) = 4.25 месяцев.




## 6. Constraints (Ограничения)

*[Укажите ограничения, которые могут повлиять на стоимость и оценку рисков. Пример: необходимость интеграции с другими системами, следование стандартам, использование определенных технологий и т.д.]*

- Заказчик может потребовать развертывание системы на своих вычислительных ресурсах. В таком случае, риск отказа системы становится выше, но ложится на плечи поставщика вычислительных ресурсов. Значит, такое решение снимает часть ответственности с Equilibrium inc.

- Заказчик выступает от лица государственных структур и может потребовать дополнительную защиту информации: Шифрование данных, использование сертифицированных сервисов для хранения данных. Издержки на такие технологии увеличат стоимость продукта, но окупятся за счет продажи подписки по более высокой цене.
